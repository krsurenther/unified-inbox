# UI Flow Review — 2026-07-11

**Scope:** the operator's core job — *find a customer, read the conversation, reply, move to the next one* — reviewed end-to-end against the current renderer (`src/renderer/App.tsx`, single-window, 2-column layout).

**Verdict:** the pipeline (sync → draft → approve → send) is solid, but the UI makes the operator do the machine's work: there is no search, no per-account filtering, no keyboard flow, and the highest-value context (orders) steals the conversation's space. Below: friction inventory by job step, then the redesign proposal.

---

## 1. Friction inventory

### Job step A — "Find the customer"

| # | Friction | Severity | Detail |
|---|----------|----------|--------|
| A1 | **No search at all** | 🔴 Critical | No way to find a thread by name, phone, or message text. With 224 threads the only tool is scrolling. This alone breaks "find and reply at ease". |
| A2 | **No per-account filter** | 🔴 High | 3 WhatsApp numbers are lumped into one "WhatsApp" tab; 3 Duoke shops into one "Marketplace" tab. You renamed numbers (Temerloh Shop, Mentakab Shop, Warehouse) precisely because they are different businesses — but you can't view one at a time. |
| A3 | **Filter chips wrap into a ragged block** | 🟡 Medium | 6 chips + counts wrap to 2–3 rows at 320px, worse when counts grow. Horizontal chips don't scale with channels; a vertical list does. |
| A4 | **Sort row spends prime space** | 🟢 Low | A full-height row for a Newest/Oldest toggle used occasionally. |
| A5 | **Rows are hard to scan** | 🟡 Medium | No avatar/color identity; channel is a small text chip below the preview; the eye has nothing to anchor on when scrolling fast. |

### Job step B — "Read the conversation"

| # | Friction | Severity | Detail |
|---|----------|----------|--------|
| B1 | **Order card pushes the chat down** | 🔴 High | Orders render *above* history inside the conversation column. A 2-item order eats ~40% of the pane; the actual conversation is squeezed below. Duoke itself puts this in a right panel — correct instinct. |
| B2 | **No customer context** | 🟡 Medium | Nothing about who this is: total orders, phone, other threads from the same person on other channels. |
| B3 | **Raw platform codes render as text** | 🟡 Medium | `[10007]`, `[10015]`, `[3]`, `[interactive]`, `[heart emoji]` appear in previews and bubbles. Meaningless noise to the operator. |
| B4 | **Full datetime under every bubble** | 🟢 Low | `11/07/2026, 19:20:53` repeated under every message is heavy; day separators + short times read faster. |

### Job step C — "Reply"

| # | Friction | Severity | Detail |
|---|----------|----------|--------|
| C1 | **No keyboard flow** | 🔴 High | Zero shortcuts. Every reply = mouse to thread → mouse to textarea → mouse to Approve & Send. The operator loop (dozens of replies/day) should be: `↑↓` threads → type/edit → `⌘⏎` send → auto-advance. |
| C2 | **No "next unanswered" flow** | 🟡 Medium | "Needs reply 81" tells you the queue size but there's no way to *work the queue* — you re-scan the list after every send. |
| C3 | **Permanent safety banner** | 🟢 Low | "🔒 Auto-send OFF · human approval required" renders on every thread forever. Right message, wrong prominence after day one. |
| C4 | **No canned replies** | 🟢 Low | Common answers (bank details, opening hours, warranty steps) are retyped or wait for the AI. |

### Job step D — "Move on / keep house"

| # | Friction | Severity | Detail |
|---|----------|----------|--------|
| D1 | **Done/Mute only inside the thread** | 🟡 Medium | Triage actions require opening the thread first; no hover/swipe actions on rows, no bulk. |
| D2 | **Notification doesn't align the list** | 🟢 Low | Clicking a notification selects the thread, but the active filter may not contain it — detail shows a thread the list doesn't. |
| D3 | **Webstore tab is a dead end** | 🟢 Low | Permanent "Webstore 0" for a channel that doesn't exist yet. |

## 2. Redesign proposal

### Layout: 2 columns → 4 zones (user preference: channels on the side)

```
┌──────────────┬──────────────┬────────────────────────────┬───────────────┐
│ NAV RAIL     │ THREAD LIST  │ CONVERSATION               │ CONTEXT       │
│ (~216px)     │ (~300px)     │ (flex)                     │ (~300px,      │
│              │              │                            │  collapsible) │
│ 🔍 Search ⌘K │ [filter box] │ name · channel · actions   │ Customer card │
│              │ sort ▾       │ ────────────────────────── │  orders count │
│ 📥 Needs  81 │              │ day separator              │  phone/id     │
│ ⬜ All    224 │ ● kang    4m │   bubbles (full height)    │ ───────────── │
│ ✓ Done    16 │ ○ lhywyy  8h │                            │ Order card    │
│ ──────────── │ ○ 0126…  20h │                            │  #50654…      │
│ WHATSAPP     │              │                            │  Samsung TV   │
│ ● Temerloh 12│              │                            │  MYR 1187.81  │
│ ● Mentakab  8│              │ ────────────────────────── │  tracking     │
│ ● Warehouse 3│              │ ✨ AI draft · gemini        │               │
│ MARKETPLACE  │              │ [textarea]                 │               │
│ ◆ Lazada   6 │              │ ↻ ⌘R      ✓ Approve ⌘⏎    │               │
│ ◆ Shopee   2 │              │                            │               │
│ ◆ TikTok   1 │              │                            │               │
│ ──────────── │              │                            │               │
│ ⚙ Settings   │              │                            │               │
│ ✆ WhatsApp   │              │                            │               │
└──────────────┴──────────────┴────────────────────────────┴───────────────┘
```

- **Nav rail** = triage states (Needs reply / All / Done) + **one row per added account** (each WA number by its custom label with live status dot + unread count; each Duoke shop by platform; Webstore appears only when connected). Channel rows filter the list. This is the user-requested "filters at the side, see messages from each channel".
- **Thread list** gets a **search box** (name / phone / message text) and a compact sort menu.
- **Conversation** gets its full height back — orders move to the context panel.
- **Context panel** (collapsible, remembers state): customer card + order cards. On <1100px it overlays instead of docking.

### Interaction upgrades

1. **Search** (`⌘K` or `/` focuses it): matches customer name, phone/ID, and message bodies. Store-side `searchThreads(q)` (SQL LIKE over customers + messages) — fast at this scale.
2. **Keyboard loop**: `↑/↓` move thread selection · `⌘⏎` Approve & Send · `⌘R` Regenerate · `E` Done · `Esc` back to list. Shortcuts shown in tooltips.
3. **Work-the-queue**: after Send/Done in the *Needs reply* view, auto-advance to the next unanswered thread (toggleable).
4. **Row quick actions** on hover: ✓ Done · 🔇 Mute, no need to open the thread.
5. **Humanize platform codes**: map `[10007]`-style Shopee/TikTok codes and `[image]`/`[interactive]` placeholders to labeled chips (🏷️ Sticker, 🖼️ Photo, 📋 Card) in previews and bubbles.
6. **Quieter chrome**: safety note becomes a small footer line; day separators in history; short bubble times.
7. **Notification click** also switches the filter to *All* so list and detail always agree.

### Explicitly kept

- Dark theme, current palette and type stack (system font) — this is a refinement, not a rebrand.
- Draft-first composer (AI suggestion pre-filled, human approves) — unchanged pipeline, all IPC intact.
- Anti-ban pacing UX (send ETA in the button) — unchanged.

## 3. Phased build plan

| Phase | Contents | Effort | Risk |
|-------|----------|--------|------|
| **R1 — Find** | Channel nav rail (per-account rows + counts + status dots) · search box + `searchThreads` in store · sort → menu · row identity (avatar initials + channel color) | ~1 day | Low — additive; filter semantics extend the existing `Filter` union |
| **R2 — Read** | Context panel (customer + orders) · collapsible + responsive · history day separators · humanized code chips | ~1 day | Low |
| **R3 — Reply** | Keyboard shortcuts · auto-advance queue · row hover actions · footer safety note | ~½ day | Low |
| **R4 — Polish** | Canned replies (settings-stored snippets) · bulk select · notification/filter alignment | later | — |

Each phase ships behind the existing test gate (vitest + tsc baseline 4) and is verified live before commit.

## 4. Revision 2 — owner feedback (2026-07-11)

Accepted changes to the proposal above:

1. **Collapsible nav rail** — folds to a 56px icon-only strip (chevron at the rail foot; state persisted). Unread count moves onto the inbox icon as a badge; channel rows collapse to their status dots.
2. **Richer customer panel** — customer card gains: avatar, "since" date, stat row (orders · lifetime MYR spent · returns), buyer ID, current assignee, other channels this customer uses, and a shared **notes** card.
3. **Auto-advance** stays in the plan but explicitly **optional and OFF by default** (explained to owner: after Send/Done in the Needs-reply queue, the next unanswered thread opens automatically).
4. **AI generation is on-demand only** — ⚠ behavioral change to the pipeline: `InboxService.maybeDraft` no longer auto-drafts on every inbound. The composer is empty until the operator presses **Generate with AI (⌘G)**. Saves tokens on every conversation that doesn't need AI help. (Config flag `autoDraft` keeps the old behavior available; default false. The "draft ready" row indicator only appears for generated drafts.)
5. **Staff assignment** — push a conversation to a person: assignee dropdown in the conversation header + row hover action; "Assigned to me" queue in the rail; audit line in the chat ("Assigned to Farah by Suren"). Staff list is a simple settings-stored list (no accounts/auth in v1); `threads.assignee` column + store methods.
6. **No emoji in UI chrome** — consistent inline-SVG line-icon set (search, inbox, check, phone, bag, globe, gear, sparkle, send, refresh, user, mute, lock, tag, image, box, clip, info). Emoji remain only inside chat content.

Phase impact: R1 gains the collapsible rail; R2 gains the richer customer card + notes; R3 swaps auto-draft for the Generate button and adds assignment. Everything else unchanged.
