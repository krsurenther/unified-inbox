# Unified Inbox — Full Review (2026-07-10)

**Scope:** entire app at commit `3094806` (Phases 0–5a) — core pipeline, channel adapters,
Electron main, MCP server, renderer UI — plus a live-UI inspection with real data
(180 threads, 122 WhatsApp, 52 marketplace, both WA numbers linked).
**Method:** three independent deep-read passes (core/data · channels/main/MCP · UX/DX)
merged with a maintainer pass and on-screen verification. Findings are deduplicated and
re-ranked; every item was verified against the code (file:line) or observed live.

---

## Verdict

The **architecture is genuinely good** — the swappable-adapter and human-in-the-loop
claims are structural facts, not README aspirations, and the anti-ban guard is substance.
What's missing is the layer between "working pipeline" and "daily driver": **operational
resilience** (a handful of hang/drop/race paths, all fixable in hours) and the **triage
loop** (read/done/notify/timestamps) that turns a message viewer into a support tool.
The single biggest product gap is **draft quality**: gemma3:4b on Malay/Manglish retail
chat produced a literal `"Okay"` for a TV-supply thread. The pipeline is sound; the model
and prompt need investment.

## Scorecard

| Area | Grade | One-liner |
|---|---|---|
| Architecture & seams | **A** | Core is Electron-free, adapters/providers truly swappable, one typed IPC contract |
| Safety (human-in-loop, audit, anti-ban) | **A−** | Only-send-path + audit log + caps/pacing/kill switch; kill switch not persisted |
| Data integrity | **B−** | Idempotency designed right, but busy-timeout gap can drop inbound; draft race mis-audits |
| Channel resilience | **C+** | No reconnect, silent JWT rot, no shutdown path, hangable send driver |
| Inbox UX (as a daily driver) | **C** | No notifications, no read/done, no timestamps, edits lost, opens at top |
| AI draft quality | **C** | Pipeline solid; model too weak for BM/Manglish; no business context |
| Test discipline | **B+** | 60 green, seams injectable; renderer/manager/MCP untested |
| DX & packaging | **C+** | Dev-only run story; cwd-dependent paths break packaging; scripts/ cluttered |

## What's genuinely strong

- `src/core` has zero Electron/vendor imports; every seam (clock, random, sleep, clients,
  store engine) is injected — 60 tests run without a browser. The official-API swap story is credible.
- `InboxService.approveAndSend` is the **only** outbound path; `send_audit` is immutable,
  records the approver, and is *load-bearing* — the anti-ban cap is derived from it, so the
  guard survives restarts.
- The Duoke send driver has real defense-in-depth (Vuex conversation verify before/after
  compose + post-send delivery confirmation via the read API) — learned from the mis-send incident.
- Ingest idempotency via partial unique index + `INSERT OR IGNORE` with `changes`-driven
  side effects, covered on both push and pull paths.
- Electron baseline is sane: contextIsolation on, typed `contextBridge` only, real CSP.
- The 3s-poll-vs-edit hazard was anticipated: polling never touches `draftBody`.
- Dark theme fundamentals pass WCAG AA (muted text 5.7:1, body 14.6:1).

## Findings

### P0 — fix before anything else

1. **No `busy_timeout` on any SQLite connection → inbound messages can be DROPPED.**
   `InboxStore.ts:39` never sets `busy_timeout` (default 0); the app and the MCP server
   share the DB file. MCP even opens **read-write and runs schema DDL** at boot
   (`server.ts:27`, `InboxStore.ts:43`). A `SQLITE_BUSY` inside `ingest`'s
   `recordInbound`/`upsertCustomer` is **not** caught (only `generateDraft` is), propagates
   through `void this.handleIncoming(msg)` (`WhatsAppAdapter.ts:94`) as an unhandled
   rejection, and the customer message is silently lost until a later backfill. Observed
   "database is locked" bursts confirm real contention.
   **Fix:** `PRAGMA busy_timeout = 5000` (+ WAL) in the `InboxStore` constructor
   (not schema.sql — the MCP process can't always find it: `resolve(process.cwd(), …)`);
   add a `readOnly` store option (`new DatabaseSync(path, {readOnly:true})`, skip DDL) for
   the MCP server; wrap store writes in a tiny retry.

2. **Staff will not know a customer wrote in.** Zero `Notification`, dock badge, or sound
   in the codebase. For a support tool, response latency is the product; a row silently
   appearing in a backgrounded window is the failure mode.
   **Fix:** on `inserted===true` in `ingest`, main fires `new Notification` (click →
   focus thread) + `app.dock.setBadge(needsReplyCount)`. ~30 lines.

### P1 — correctness & resilience

3. **`approveAndSend` marks the wrong draft as sent.** `InboxService.ts:176` looks up
   `getLatestDraft` **after** the send, and sends are slow by design (2.5–15s pacing,
   ~4s Duoke confirm). An inbound arriving mid-send creates a fresh draft that gets
   `status='sent'` + the audit `draft_id` — a draft no human ever saw — while the approved
   one stays `suggested`. **Fix:** capture the draft (better: have the renderer pass the
   explicit `draftId`) *before* `adapter.send()`.

4. **Human draft edits are never persisted — and can be clobbered mid-typing.**
   `draftBody` is renderer state only; no save IPC exists; `edited`/`approved`/`discarded`
   statuses (types.ts:49, schema.sql:65) are dead. Switch threads and back → edit gone.
   Worse, draft-on-open regenerate (`App.tsx:70-80`) `setDraftBody(d.body)` unconditionally —
   it overwrites text the user typed while "Drafting…". **Fix:** debounced
   `inbox:updateDraft` (status `edited`); a `dirtySinceRegen` flag so regenerate never
   clobbers; ingest/sync skip regeneration when the live draft is `edited`.

5. **`loadConfig()` is dead code — the whole config story is unwired.** Both `bootCore`
   (`main/index.ts:59`) and the MCP server hardcode `AppConfigSchema.parse({...})`;
   `config.local.json`, `UNIFIED_INBOX_CONFIG`, and (in-app) `UNIFIED_INBOX_DB` do nothing.
   Operator cannot change systemPrompt, per-channel provider, `dailyCap`, or WA labels
   without editing source; app and MCP already disagree (channels map). The zod-default
   systemPrompt also lacks config.example's "reply in the customer's language / don't
   invent" lines. **Fix:** `loadConfig()` from `app.getPath('userData')/config.json` with
   env override; MCP derives the same path; drop `config.example.json` from the runtime
   fallback chain (its `defaultProvider:"echo"` would silently downgrade drafts).

6. **Hung Ollama stalls everything; the 60s interval stacks syncs.**
   `OllamaProvider.ts:49` fetch has no `AbortSignal` (hangs up to ~300s); `syncChannel`
   drafts serially; `main/index.ts:207` fires `void syncDuoke()` every 60s with **no
   single-flight latch** (and `bootDuoke`'s guard flips only after `listShops()` resolves —
   reentrancy can double-register adapters). Result: stacked duplicate syncs, duplicate
   drafts, amplified DB contention. **Fix:** `AbortSignal.timeout(30_000)` + an
   `inFlight` latch (or chained `setTimeout`).

7. **No shutdown path.** Nothing calls `service.stop()` / `store.close()` / clears the
   interval on quit (`main/index.ts:220`). Orphaned Chrome-for-Testing per linked number
   (known symptom) — and a survivor holding the LocalAuth profile lock can make the *next*
   launch's `initialize()` fail. **Fix:** `before-quit` handler: clear interval,
   `service.stop()` (reaches every adapter), close send driver + DB, then `app.exit(0)`.

8. **A dropped WhatsApp number stays dead — and Connect silently no-ops.** On
   `'disconnected'` only state changes; `e.adapter` stays set so `connect()`'s
   `if (!e || e.adapter) return` guard (`WhatsAppManager.ts:84`) blocks retry. No
   auto-reconnect; `auth_failure` is swallowed with no manager callback (stuck
   "connecting"). **Fix:** on disconnect, clear `e.adapter`, unregister, `stop()`, then
   capped-backoff reconnect; surface `auth_failure`.

9. **Kill switch is in-memory — an app restart silently re-arms sending**
   (`WhatsAppGuard.ts:37`). The operator engages it exactly when a number is at risk;
   a crash/relaunch quietly resumes. **Fix:** persist in a tiny `settings` k/v table in
   the existing store; read at construction; show "restored: ON".

10. **Duoke send can hang the UI forever.** `DuokeSendDriver.ts:76-83` `cmd()` promises
    resolve only on a matching CDP response — no timeout, no ws `close`/`error` rejection.
    Duoke quits mid-send → the IPC promise never settles → Send stuck until app restart.
    **Fix:** reject all pending on close/error + ~10s per-command deadline (safe: audit is
    written only after success).

11. **Duoke JWT rot is silent and unrecoverable without a restart.** `refreshToken()` is
    never called; `expiresAt` decoded then discarded; `health()` only checks non-empty;
    no `duoke:*` IPC exists — marketplace threads just quietly stop updating. **Fix:**
    auth-failure → one `refreshToken()` retry inside `DuokeClient.call`; add channel-health
    IPC + a Marketplace-tab banner ("Duoke session expired — log in again").

12. **No triage loop — the inbox can only grow.** `thread.status`
    (`open|snoozed|closed`) has no setter, no IPC, no UI; unread clears only on *reply*
    (`InboxStore.ts:142`), never on read (verified live: badge survives opening).
    At 180 threads nobody can tell what still needs work. **Fix (minimal loop):**
    mark-read on open, "Needs reply" default filter (last message inbound), one-click
    Done (status→closed) + Open/Done toggle.

13. **Conversation opens scrolled to the oldest message** — no `scrollTo`/ref in the
    renderer at all; the message being answered is always off-screen (verified live; the
    newest message also renders clipped under the composer). **Fix:** scroll-to-bottom on
    selection + stick-to-bottom when near-bottom; bottom padding on the history container.

### P2 — product correctness & experience

14. **Pacing runs inside the awaited IPC** — Send goes dead up to ~15s with no
    explanation, and the single global `busy` flag freezes *every* thread's buttons.
    No hourly burst cap either: 200/day is burnable in ~10 min of rapid approvals, and two
    quick approvals can both pass `check()` before either audit row lands. **Fix:**
    per-number send queue in main returning `{queued, etaMs}` immediately + `send:update`
    events; per-thread busy map; second `SendPolicy` window (e.g. 20/hour — same
    `countSendsSince` mechanism); queue serialization kills the check/pace race for free.

15. **Pull-path drafts go stale (push/pull asymmetry).** `syncChannel` only drafts when
    there's no live draft (`InboxService.ts:141`) — new marketplace inbound never refreshes
    a `suggested` draft, so the operator can approve an answer to a days-old question;
    `ingest` (push) always regenerates. **Fix:** one shared `maybeDraft(threadId,{newInbound})`
    rule — regenerate on new inbound unless the draft is human-`edited`.

16. **Persistent LLM failure is invisible.** All three draft sites swallow errors; with
    Ollama down every thread just says "No draft yet" — indistinguishable from
    nothing-to-draft. **Fix:** record last draft failure per thread (or a `failed` draft
    row) → composer banner + provider health dot.

17. **`status@broadcast` (and newsletters) leak into the inbox on the live path** —
    `handleIncoming` (`WhatsAppAdapter.ts:114`) filters only `@g.us`, while `listThreads`
    filters `@broadcast` (the asymmetry proves it). Verified live: a "status" thread with
    12 unread + `[video]` previews at the top, drafts generated per story. **Fix:** shared
    guard — skip `status@broadcast` and `/@newsletter$/` in both paths.

18. **Empty-body fallthrough.** `normalize.ts:49` consults `MEDIA_LABEL` only when
    `hasMedia` — non-media typed messages (`location`, `order`, `buttons_response`…)
    normalize to `''`: stored as blank bubbles (observed: thread previewing "—"), and
    ingest still drafts on an empty customer turn; the map's own `location`/`vcard` labels
    are unreachable. **Fix:** `body || MEDIA_LABEL[type] || (type!=='chat' ? `[${type}]` : '')`
    and skip drafting when the stored body is empty.

19. **NULL `channelMessageId` silently disables dedupe** — the partial unique index only
    engages when the id is present; an id-less Duoke payload would re-insert on every 60s
    sync forever. **Fix:** synthesize a deterministic fallback key
    (`hash(direction|body|timestamp)`) in `recordMessage`/`recordInbound`.

20. **Residual send-driver TOCTOU.** Verify → set-text → Enter are three separate CDP
    round-trips (`DuokeSendDriver.ts:141-158`); the user can switch conversations between
    them, and `pressEnter` re-queries the selector (and its boolean result is ignored).
    Given the history, close it fully: collapse check+set+Enter into **one**
    `Runtime.evaluate` page-turn that throws on mismatch.

21. **Demo scaffolding ships in production.** Three fake customers seed any empty real DB,
    count into "All" (180 ≠ 52+122), leak into MCP `list_threads` (verified), and
    "✦ Simulate incoming" sits in the header one click from polluting the store. **Fix:**
    one `UNIFIED_INBOX_DEMO` flag gates FakeAdapter, seeding, and the button.

22. **Thread rows carry no time, no channel color, no direction.** Verified at 180
    threads: rows are visually identical, staleness unanswerable, and the preview doesn't
    distinguish "customer said X" from "we replied X". **Fix:** relative timestamp
    right-aligned, per-kind chip color (WA green / marketplace orange), `You:` prefix on
    outbound previews.

23. **Consequential errors are a truncated 4.5s toast.** `.split(': ').pop()` strips
    meaning ("Ollama 500: …" → "chat request failed"); cap-reached / kill-switch reasons
    vanish in seconds with no trace on the thread. **Fix:** persistent inline error strip
    in the composer; toasts for successes only.

24. **Non-customer chats are ingested and drafted like customers** — supplier threads
    ("Ada supply tv ni" / "mark as delivered") and status get Ollama drafts; wasted cycles
    + mis-send risk if staff blindly approve. **Fix:** per-thread mute/exclude ("not a
    customer") that suppresses drafting + needs-reply counting.

### P3 — hygiene & polish

25. `send_audit` lacks a `(channel_id, sent_at)` index — `countSendsSince` full-scans on
    every send check and 15s guard poll.
26. Missing Duoke `lastMessageTimestamp` becomes `1970-01-01` and sinks the thread
    (`duoke/normalize.ts:124`); make `last_message_at` updates monotonic (`MAX(...)`).
27. Unknown provider id silently falls back to `defaultProvider` (`LlmRouter.ts:19`) —
    the exact knob config exists for fails without a trace; and the renderer's
    `providerId==='echo'` placeholder heuristic regenerates + inserts a new draft row on
    *every open* under a default config. Warn on unknown ids; mark placeholders explicitly.
28. IPC handlers forward renderer input unvalidated (`body` can be a non-string —
    whatsapp-web.js accepts structured content); `win` is never nulled on `'closed'`, so on
    macOS every WA event after window-close throws "Object has been destroyed"
    (`main/index.ts:146-167,202`). Five lines of type/length guards + a `closed` handler.
29. Zero keyboard support (no Cmd+Enter, no list navigation, no autofocus);
    `window.confirm` blocks the renderer with OS chrome.
30. Detail header shows the raw `@lid` number; conversation timestamps are verbose
    (`06/07/2026, 09:01:14`) — relative + day-grouping reads better.
31. Packaging blockers: `.wwebjs_auth` at `process.cwd()` (`main/index.ts:200`) and
    `db/schema.sql` resolved from cwd (`InboxStore.ts:39`) both break a packaged app
    (cwd=`/`); menu bar says "Electron"; no single-instance lock (second instance = SQLite
    contention); `sandbox:false` only because the preload is ESM — emit CJS and re-enable.
32. `scripts/` holds ~15 one-shot probes (duoke-*, net-test, wa-diag…) — sweep into
    `scripts/diagnostics/` with a one-line README; add `lint`/`format` scripts.
33. `listThreads` is an N+1 (4 extra queries per thread) polled every 3s on the sync
    main-process connection — one JOIN; then push `inbox:changed` events from ingest/sync
    and demote the poll to a slow fallback.
34. No schema versioning — `CREATE TABLE IF NOT EXISTS` silently no-ops future column
    changes; stamp `PRAGMA user_version` + numbered migrations in the store constructor.

## Draft quality (the value prop)

Observed live: for a Malay TV-supply thread, the draft was **"Okay"**. The pipeline
(history → system prompt → nudge) is correct; the model is the ceiling.

1. **Try a stronger local model first** (free, one env var): `qwen2.5:7b` or
   `llama3.1:8b` — both markedly better at Bahasa/Manglish than gemma3:4b.
2. **Enrich the system prompt with business facts** (config-driven once #5 lands): shop
   name, tone, delivery/warranty/COD policies, common answers. Biggest cheap win.
3. **Cloud provider for quality** when justified: a ~40-line `ClaudeProvider` behind the
   existing interface, key via `.env`, switched per channel in config.
4. Wire `draftReplyStream` (interface already has it) so long drafts appear progressively.
5. Unify the two diverging systemPrompts (zod default vs config.example).

## Recommended build order

**Round 1 — data safety & awareness (≈ a day):**
busy_timeout + read-only MCP store + write retry (#1) · notifications + badge (#2) ·
draft-race fix (#3) · Ollama timeout + single-flight interval (#6) · shutdown path (#7) ·
kill-switch persistence (#9) · status@broadcast filter (#17) · empty-body skip (#18) ·
**disconnect purges the number's inbox data** (owner request 2026-07-10: unlinking a
WhatsApp number deletes its threads/messages/drafts/customers; `send_audit` is KEPT so the
rolling cap can't be reset by disconnect→reconnect; update the unlink confirm copy, which
currently promises history is kept; re-link re-imports the recent window anyway).

**Round 2 — the triage loop (the product leap):**
mark-read + Needs-reply filter + Done (#12) · scroll-to-bottom (#13) · draft-edit
persistence + no-clobber (#4) · timestamps/colors/You: in list (#22) · per-thread busy +
pacing countdown via send queue (#14) · error surfacing: LLM health, Duoke token, WA
reconnect banners (#11, #16) · WA auto-reconnect (#8).

**Round 3 — config, hardening, quality:**
wire `loadConfig()` (#5) · demo gating (#21) · hourly burst cap + business-hours window ·
send-driver single-evaluate + cmd timeouts (#10, #20) · dedupe fallback key (#19) ·
draft-quality work (model swap + business prompt) · router warnings (#27).

**Round 4 — daily-driver packaging & scale:**
userData paths + electron-builder + single instance + login item (#31) · push events +
JOIN (#33) · schema versioning (#34) · keyboard shortcuts (#29) · tests: renderer
draft-preservation, WhatsAppManager state machine, MCP round-trip · scripts cleanup (#32).

**Phase 2c (webstore adapter)** slots naturally after Round 2 — the adapter seam is ready.
